제목: Strip Calendar(메인 화면) 주<->월 전환 구조 검토 요청

1. 배경
- 메인 Todo 화면에 사용할 `strip-calendar`를 설계 중입니다.
- 요구사항:
1) Weekly: 좌우 스와이프 또는 `< >`로 주 이동
2) Monthly: 위아래 스와이프로 무한 스크롤
3) 주<->월 전환은 하단 바 스와이프로만 (드래그 리사이즈 없음)
4) 메인 화면이므로 최대한 가볍게

2. 핵심 고민
- “보이지 않는 월 UI를 숨겨두면 메모리/성능 문제가 생기지 않는가?”
- 특히 주<->월 전환 애니메이션 시, 숨겨진 월/주 컴포넌트가 계속 살아있으면 느려질 가능성 우려

3. 현재 확정한 정책
- Monthly 리스트 구현체는 `@shopify/flash-list` 필수
- `FlatList`는 월간 메인 경로에서 사용하지 않음
- Dot 정책:
1) 카테고리 중복 제거 후 표시
2) 최대 3개 dot
3) 4개 이상은 작은 `+` overflow 표시
- 로컬 세팅 반영:
1) `language`에 따라 월/요일 라벨 변경
2) `startDayOfWeek`(sunday/monday)에 따라 주 시작일 변경
- 데이터 연동:
1) 현재는 UI shell 우선
2) 일정/반복 조회는 Phase3의 통합 경로 확정 후 adapter로 연결 예정

5. 내가 제안하는 구현안 (초안)
- 기본 원칙:
1) 느려지는 건 데이터보다 UI 트리이므로, 비활성 모드 UI는 unmount
2) 데이터는 캐시(Map/Zustand)로 유지 가능
- Weekly 모드:
1) Prev/Current/Next 3주(총 21칸)만 렌더
2) 좌우 스와이프/화살표로 앵커 주 이동
- Monthly 모드:
1) FlashList로 주(row) 단위 세로 무한 스크롤
2) 화면 기준 6주 높이, 나머지는 virtualization/recycle
- 모드 전환:
1) 전환 중 필요한 최소 애니메이션만 수행
2) 전환 완료 시 비활성 모드 컴포넌트는 즉시 unmount
- 계산 최적화:
1) dot/event 계산은 렌더 중 매번 하지 않음
2) 사전 계산/메모화된 결과만 DayCell에 전달
3) DayCell props 최소화 + memo

6. Opus에게 검토 받고 싶은 포인트
1) 주/월 렌더러를 분리하고 상호 unmount하는 구조가 최적안인지
2) 월간 FlashList를 “주(row) 단위”로 구성하는 것이 가장 안정적인지
3) 모드 전환 애니메이션 시점(겹침 구간/언마운트 타이밍) 권장 패턴
4) 성능 기준선 제안:
   - 장시간 스크롤 후 메모리 plateau 유지
   - 주 모드에서 셀 수 상수(21칸) 유지
   - 월 모드에서 화면 밖 셀 recycle 확인
5) Phase3 통합 시 strip-calendar adapter 인터페이스(입력/출력) 권장 시그니처
